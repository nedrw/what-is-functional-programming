# 函数式编程（FP）演示

四个示例，分别展示 Rust 中函数式编程的一个核心面。

---

## 背景：为什么需要函数式编程

### 问题

命令式代码的核心模式是：

```rust
let mut result = vec![];
for item in list {
    if condition(item) {
        result.push(transform(item));
    }
}
```

这段代码没问题。但当你面对复杂的业务逻辑时，这种写法暴露出一个问题：**你读的不是"这段代码要做什么"，而是"这个变量每一步怎么变"**。每一行都可能修改任意状态，所以理解一个函数 = 在脑子里模拟整个执行过程。

### 解决方案

函数式编程的回答是：**让数据流显式化，消除隐式的状态变更。**

三个核心约束：

1. **纯函数**：同样的输入 → 永远同样的输出，不打日志、不写文件、不改全局变量。
2. **不可变性**：数据创建后不修改，变换产生新数据。
3. **组合**：小函数通过标准接口（`map`、`filter`、`fold`）拼成大管道，而不是嵌在循环体里。

```rust
// 改写成函数式的管道：
list.iter()
    .filter(condition)
    .map(transform)
    .collect()
```

这段代码声明了三件事：要筛选、要变换、要收集。每一步做什么是独立的——`filter` 看不懂 `map` 的逻辑，`map` 也不需要知道数据从哪来。你可以只关心当前这一步，而不需要理解整个循环体。

### 这不只是语法糖

区别不在"几行代码"，而在**谁负责控制流**：

| 风格 | 控制权在 | 读代码的人需要 |
|------|---------|--------------|
| 命令式 for 循环 | 你的手写逻辑 | 模拟每一次迭代和状态变更 |
| 函数式组合子 | `filter` / `map` 等标准库 | 只看当前变换的职责 |

用 for 循环时，你既要描述"遍历"、又要描述"条件"、又要描述"变换"——三个关注点挤在一个块里。用组合子时，每个关注点被拆成了独立的函数，再交给标准库去编排执行顺序。

### Rust 为什么适合 FP

Rust 不是 Haskell，不会强迫你全程纯函数。但它提供了足够的基础设施，让你可以在**最合适的局部**使用 FP 风格：

- 变量默认不可变（`let` vs `let mut`）
- 迭代器是零开销抽象（组合子在编译期被内联，不会比手写循环慢）
- `Option` / `Result` 是代数类型，强制你处理所有分支
- 所有权系统天然消灭了"共享可变状态"——这正是并发编程最大的敌人

下面四个示例将展示这些能力如何在真实场景中落地。

---

## 示例 1: 数据筛选 (`data_filtering`)

```sh
cargo run --example data_filtering
```

**FP 要点：迭代器组合子**

```rust
orders.iter()
    .filter(|o| o.category == "food")   // 筛选
    .map(|o| o.amount)                  // 变换
    .sum();                             // 聚合
```

整个过程无 `mut`、无 for 循环、无中间变量。数据像流水线上的零件依次经过每个工位（组合子），最终产出结果。`filter` 只管"要不要"、`map` 只管"怎么变"，各组合子职责单一，可以任意重新排列组合。

`fold` 是更底层的组合子——上面的 `sum`、`filter`、`map` 都可以用 `fold` 实现，但它也暴露了"需要 mutable 累加器"这个例外（因为累加器是临时内部状态，不泄露到函数外部，所以你依然可以把它看作纯函数——同一个输入永远产生同一个输出，只是内部实现借用了临时可变性）。

## 示例 2: 并行计算 (`parallel_compute`)

```sh
cargo run --example parallel_compute
```

**FP 要点：不可变性 + 无共享状态 = 自动并行**

```rust
// 串行
data.iter().map(|x| x * x).sum();

// 并行（唯一改动！）
data.par_iter().map(|x| x * x).sum();
```

为什么可以一行改动就安全并行？因为 `map` 和 `sum` 都是**纯函数**：每个闭包只依赖于输入 `x`，不修改全局状态。rayon 可以把数据分到多个线程各自计算平方，最后合并结果，无需锁或 `Arc<Mutex<>>`。

这不是"编译器自动并行化"——Rust 没有做这个。而是 **FP 风格天然地将"做什么"与"怎么做"分离**，使得库可以透明地改变执行策略（并行/惰性/异步），而业务代码无需改动。

## 示例 3: 错误处理链 (`error_handling`)

```sh
cargo run --example error_handling
```

**FP 要点：Option / Result 作为单子（Monad）**

```rust
parse_number(input)                 // Result<i32, String>
    .and_then(require_positive)     // 如果 OK 就继续，否则短路
```

`and_then` 的本质：一个可能失败的计算结束后，如果成功就接上另一个可能失败的计算，如果失败就直接跳过后续。这种"条件继续"的逻辑被封装在 `and_then` 里面，业务代码只要声明"先做什么，再做什么"。

`?` 运算符做了同一件事，但语法上是"提前 return 错误"。两种方式等价——选择哪种取决于上下文：线性的短链用 `and_then` 更函数式，需要分支或多步骤时 `?` 更清晰。

---

## 示例 4: 状态机 (`state_machine`)

```sh
cargo run --example state_machine
```

**FP 要点：纯函数定义状态转换**

```rust
fn transition(state: State, event: &Event) -> State
```

这个签名本身就是承诺：给定当前状态和事件，返回新状态。没有内部可变状态、没有隐式全局变量、没有 I/O。测试只需构造输入、断言输出，不需要 mock 任何东西。

执行过程是 `fold` 的另一种体现：初始状态 + 事件列表 → 经过一系列纯函数转换 → 最终状态。

---

## 贯穿线索

四个示例共享同一个模式：**数据流形状决定代码结构**。

| 示例 | 数据流 | 核心组合子 |
|------|--------|-----------|
| data_filtering | 集合 → 子集 → 聚合值 | filter, map, fold |
| parallel_compute | 同上，但并行 | par_iter, map, sum |
| error_handling | 线性链，带短路 | and_then, `?` |
| state_machine | 状态 + 事件序列 → 最终状态 | fold, match |

每个示例中，你都看不到 for 循环里堆砌 if-else。取而代之的是：先声明要做什么变换，再把它组合成管道。代码输出什么是由遍历的元素决定的，而不是由你在循环体里写的控制流决定的。

**这就是函数式编程的核心：用组合替代命令，用数据流替代控制流。**
